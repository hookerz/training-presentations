What is Web Accessibility?

  • Web accessibility means that people with disabilities can use the Web.
  • Web accessibility can also benefit people without disabilities in that accessible sites are more flexible

In the most simple terms, web accessibility means that people with disabilities can use the Web. More specifically, they can perceive, understand, navigate, and interact with the Web. Web accessibility encompasses all disabilities that affect access to the Web, including visual, auditory, physical, speech, cognitive, and neurological disabilities. Web accessibility also benefits people without disabilities in that accessible sites are flexible even for people with slow Internet connections, people with "temporary disabilities", such as a broken arm, and people with changing abilities due to aging, people with low literacy or not fluent in the language.

Why Web Accessibility is Important

  • The Web is an increasingly important resource in many aspects of life: education, employment, government, commerce, healthcare, recreation, and more.
  • The Web also offers the possibility of unprecedented access to information and interaction for many people with disabilities.
  • It is essential that the Web be accessible in order to provide equal access and equal opportunity to people with disabilities

The Web is an increasingly important resource in many aspects of life. Technology is being adopted rapidly in education, employment, government, commerce, healthcare, and recreation, which is great! But what this means is that it is essential for the Web to be equally accessible to everyone, including people with disabilities. This importance is especially underscored by the fact that accessibility barriers to print, audio, and visual media can be much more easily overcome through web technologies.

Web Content Accessibility Guidelines

The Web Content Accessibility Guidelines (WCAG) are a technical standard developed by the W3C, in cooperation with individuals and organizations around the world, with a goal of providing a single shared standard for web content accessibility that meets the needs of individuals, organizations, and governments internationally.

  • WCAG 2.0 is the current stable, referenceable version
  • It has 12 guidelines that are organized under 4 principles:
    • Perceivable
    • Operable
    • Understandable
    • Robust
  • For each guideline, there are testable success criteria, which fall under three levels: A, AA, AAA

The Web Content Accessibility Guidelines are a technical standard developed by the W3C with input from individuals and organizations around the world, with a goal of providing a common, widely-used standard for web content accessibility. WCAG is currently on version 2.0. WCAG is organized into 12 guidelines that are categorized under 4 principles.

For each guideline, there are testable success criteria, which fall under three levels: A, AA, AAA, with A meeting basic accessibility guidelines and AAA meeting the most advanced guidelines.

WCAG 2.0 at a Glance

  • Perceivable
    • Provide text alternatives for non-text content
    • Provide captions and other alternatives for multimedia
    • Create content that can be presented in different ways, including by assistive technologies, without losing meaning
    • Make it easier for users to see and hear content
  • Operable
    • Make all functionality available from a keyboard
    • Give users enough time to read and use content
    • Do not use content that causes seizures
    • Help users navigate and find content
  • Understandable
    • Make text readable and understandable
    • Make content appear and operate in predictable ways
    • Help users avoid and correct mistakes
  • Robust
    • Maximize compatibility with current and future user tools

Criteria for safe flashing content is a flash frequency of 3 times per second or fewer.

WAI-ARIA

  • The Web Accessibility Initiative (WAI) develops strategies, guidelines, and resources to help make the Web accessible
  • WAI-ARIA, the Accessible Rich Internet Applications (ARIA) suite, defines a way to make web content and web applications more accessible
  • It especially helps with dynamic content and advanced user interface controls developed with AJAX, HTML, JavaScript, and related technologies

The Web Accessibility Initiative (WAI) develops guidelines widely regarded as the international standard for web accessibility. These guidelines are known as WAI-ARIA, with ARIA meaning Accessible Rich Internet Applications. Currently certain functionality used in websites is not available to some users with disabilities, especially people who rely on screen readers and people who cannot use a mouse. WAI creates support materials to help understand and make implementing web accessibility easier for web authors, This includes defining new ways for functionality to be provided to assistive technology. It especially helps with dynamic content and advanced user interface controls developed with AJAX, HTML, JavaScript, and related technologies.

Mobile Accessibility

  • People with disabilities using computers have similar interaction limitations as people without disabilities who are using mobile devices
  • As result, there is also significant overlap between the design solutions for both
  • There are not separate guidelines for mobile accessibility — mobile is covered in existing accessibility guidelines

People with disabilities using computers have similar interaction limitations as people without disabilities who are using mobile devices; As result, there is also significant overlap between the design solutions for both. So mobile accessibility is already covered in existing guidelines.

For example, let's say that something on a webpage is conveyed solely with color. In the context of someone who is disabled, the user could be blind or colorblind and therefore perceive color incorrectly or not at all. In the context of someone using a mobile device, the user could be outside in bright light, and therefore the color could be washed out.

Making AJAX and Related Technologies Accessible

  • Many web applications developed with AJAX pose additional accessibility challenges.
  • WAI-ARIA provides a framework for adding attributes to identify features for user interaction, how they relate to each other, and their current state.
  • For example, with WAI-ARIA, developers can identify regions of pages and enable keyboard users to easily move among regions rather than having to press Tab many times
  • WAI-ARIA also includes technologies to map controls, AJAX live regions, and events to accessibility APIs, including custom controls used for rich Internet applications

Websites and web applications are increasingly using more advanced and complex user interface controls. To provide an accessible user experience, assistive technologies need to be able to interact with these controls. In particular, the use of AJAX poses additional challenges. For example, if the content of a webpage changes in response to user actions or time- or event-based updates, that new content may not be available to some people. WAI-ARIA provides a framework for adding attributes to identify features for user interaction, how they relate to each other, and their current state.

Technical Solutions

  • WAI-ARIA provides web authors with the following:
    • Roles to describe the type of widget presented, such as "menu", "slider", and "progressmeter"
    • Roles to describe the structure of the web page, such as headings, regions, and tables
    • Properties to describe the state widgets are in, such as "checked" for a checkbox, or "haspopup" for a menu
    • Properties for drag-and-drop that describe drag sources and drop targets*
    • Properties to define live regions of a page that are likely to get updates (AJAX-powered)
    • A way to provide keyboard navigation for the web objects and events

Building Accessible Applications with WAI-ARIA

The following steps are recommended when WAI-ARIA is applied to content:

  1. Use native markup when possible.
  2. Apply the appropriate roles.
  3. Preserve semantic structure.
  4. Build relationships.
  5. Set states and properties in response to events.
  6. Support full, usable keyboard navigation.
  7. Syncronize the visual interface with the accessible interface.

1. Use Native Markup When Possible

  • Use the semantic elements that are defined in the host markup language.
  • For example, with HTML, it is better to use the native checkbox than to use a div element with the role "checkbox" as these should already by accessible through the browser.
  • WAI-ARIA roles, states, and properties are best used when the markup does not support all the semantics required.

We should also use markup to convey meaning and structure. This means using HTML5 elements such as nav, header, section, footer, rather than divs. If a native element doesn't exist for what you're trying to make, WAI-ARIA roles, states, and properties can provide additional meaning (ex. role="search" to identify search functionality). Keep in mind that if you add a role attribute to an element, it will override and change the behavior of that element.

2. Apply the Appropriate Roles

  • Roles convey additional information that the assistive technologies need to anticipate the behavior of the elements inside the application, such as how to present the corresponding WAI-ARIA states and properties to the user.
  • Set roles to make sure elements behave predictably and correctly describe the behavior of each element wihin the application.
  • When a role attribute is added to an element, the semantics and behavior of the element are augmented or overridden by the role behavior.
  • Roles are element types and you must not change role values over time or with user actions.
  • If you wish to change a role, you must do so by deleting the associated element and its children and replace it with a new element with the appropriate role.

Usable keyboard navigation in a rich internet application is different from the tabbing paradigm in a static document.

The tabindex attribute enables focus to be moved via keyboard to HTML elements. Each element that needs to receive keyboard focus needs to have a tabindex attribute set. Elements with tabindex of -1 will not be added to the tab sequence but are enabled to receive keyboard focus. Elements with tabindex = 0 will be added to the tab sequence based on document order. Elements with tabindex greater than 0 will be added to the tab sequence based on the tabindex value. Avoid using this because it causes problems and is difficult to maintain.

This extension will add an accessibility audit and an accessibility properties sidebar pane in the Elements tab of your Chrome Developer Tools. To run an audit, you would just go to the Audits tab, select the Accessibility audit, and click Run. This tool is really helpful in that it returns violations then points you to the Google Accessibility Wiki where it explains the violation, and has code snippets on the correct and incorrect way to do things.

SEE is another Chrome extension that lets you view the web as someone with a visual impairment might. One in 12 men and one in 200 women are affected by colorblindness which adds up to millions of people around the world. Given these statistics, it could be helpful to know how a webpage looks to this demographic of users. The SEE extension includes settings for different kinds of colorblindness as well as other eye conditions, such as glaucoma and diabetic retinopathy.

Google Accessibility Rating (GAR)

  • GAR is a rating scale for measuring the level of accessibility of software
  • Ratings range from 1-5
  • Tests are split into "Basic" and "Advanced"
  • Achieving a level 5 rating requires user testing

The Google Accessibility Rating is a rating scale created by Google for measuring the level of accessibility for a software product. The ratings scale ranges from 1-5 with one being the most basic and 5 the most advanced.

There is a five step process in order to make sure your product meets the rating level that you're looking to achieve. The steps involve filling out a self assestment template, defining your product's use cases, performing tests, and submitting it for reviews and then approval.

Tests are split into 2 categories: Basic and Advanced. The tests are basically a series of questions that help you to evaluate if your product meets all the various WCAG guidelines that we've discussed, such as color contrast, keyboard operability, text alternatives, etc. If you're aiming for a rating level of 2 or higher, then the advanced tests also need to be performed. If you want to achieve a level 5 rating, user testing is required with people who have disabilities.
